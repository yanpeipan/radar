## Open Question Answers

**Q1: When to deduplicate?**
A: At report generation time (before clustering). Exact + MinHash dedup removes true duplicates; semantic dedup keeps different perspectives (threshold 0.92 cosine similarity, high enough to only catch near-identical content).

**Q2: How to select optimal cluster count (k) dynamically?**
A: Use `k = max(5, min(sqrt(n/2), 50))` where n = article count. This gives a reasonable range (5-50) that scales with data size without over-clustering.

**Q3: Where to store MinHash signatures?**
A: SQLite column `articles.minhash_signature` (BLOB). Queryable, fits existing schema pattern. ChromaDB metadata is not queryable, so it is not suitable.
</context>

Create `src/application/dedup.py` with the three-level deduplication pipeline.

**Level 1 — MD5 Exact Dedup:**
- Compute SHA256 of `title + first_500_chars` → `content_hash`
- Store in SQLite articles.content_hash column on first store
- Before report: SELECT id FROM articles WHERE content_hash IN (batch) → mark duplicates

**Level 2 — MinHash Approximate Dedup:**
- On article store: compute MinHash(text, num_perm=128) → store as pickled BLOB in minhash_signature
- Before report: load MinHash signatures for date range, use datasketch.MinHashLSH(threshold=0.85)
- Query LSH for near-duplicates (Jaccard >= 0.85)

**Level 3 — Embedding Semantic Dedup:**
- Use ChromaDB `articles` collection (already has embeddings)
- Fetch embeddings for all articles in date range
- Compute pairwise cosine similarity (sklearn.metrics.pairwise.cosine_similarity)
- Mark as duplicate if cosine_sim >= 0.92
